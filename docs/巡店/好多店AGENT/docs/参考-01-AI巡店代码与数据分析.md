## 2. 主链路

主入口是 `AiInspectionFacadeImpl.inspection()`，核心编排在 `AiInspectionCapturePictureServiceImpl`。

```text
策略和时段
  -> AI_INSPECTION_DATA_DEAL
  -> 区域/门店拆分
  -> AI_INSPECTION_STORE_TASK
  -> 设备抓拍
  -> 巡检 Period 与单图记录
  -> 同步或异步模型
  -> 周期聚合
  -> 人工复核或自动分流
  -> AI_INSPECTION_STORE_QUESTION
  -> aiInspection Question / QUESTION_ORDER
```

阶段入口：

1. `capturePicture`：匹配运行日、时段和基础频率；
2. `decomposeStores`：展开区域、校验创建人门店权限和门店状态；
3. `storeCapture`：处理设备、频率、营业状态、抓拍和模型；
4. `updateAiInspectionResult`：聚合图片并更新 Period；
5. `handleInspectionResultAndCreateTicket`：根据策略和置信度分流；
6. `reviewConfirm`：人工确认违规或误报；
7. `AiInspectionQuestionListener`：异步创建 Question；
8. Question 状态变化后回写 Period `ticketStatus`。

该链路跨 MySQL、Redis、RocketMQ、设备和模型，是分段事务和最终一致流程，不是单个 Service 方法内的 ACID 事务。

## 4. 策略配置事实

`AiInspectionStrategiesExtendInfo` 当前包含：

| 字段 | 代码语义 |
|---|---|
| `reviewers` | 复核人员或岗位，按门店解析 |
| `ticketCreateMode` | `0` 智能分流、`1` 全人工、`2` 全自动、`3` 不发起 |
| `autoIssueThreshold` | 智能分流自动发单阈值，默认 90 |
| `reviewThreshold` | 进入复核或自动模式的最低阈值，默认 60 |
| `ticketCreateRule` | `0` 不发起、`1` 每次、`2` 当前时段一次、`3` 当天一次 |
| `enableFrequencyScheduling` | 按连续合格次数延长抓拍间隔 |
| `failHandleType` | 不合格后的继续、时段停止或当天停止 |
| `enableDeviceOfflineProtect` | 设备异常计数、挂起、探测和恢复 |

`ticketCreateMode` 由数字常量和 DTO 注释表达，没有独立 Enum；`ticketCreateRule` 有 `TicketCreateRuleEnum`。

## 5. 调度、抓拍和模型

### 5.1 调度

`AiInspectionFacadeImpl.inspection()` 根据企业切换租户数据源并调用 `capturePicture()`。后者解析星期和时间，查询启用策略与时段，使用 `isInPeriodRound()` 判断频率，并发送 `AI_INSPECTION_DATA_DEAL`。

外部调度系统不在当前仓库，本仓库通过 SOFA Facade 暴露入口。

### 5.2 门店拆分

`decomposeStores()` 展开区域、合并直接门店、按策略创建人的可见门店过滤、排除无效门店，然后逐店发送 `AI_INSPECTION_STORE_TASK`。

**代码事实**：当前平台 AI 巡店 Scope 继承策略创建人的门店权限，不存在 AI 员工独立 Scope。

### 5.3 单店执行

`storeCapture()` 依次处理特殊检查项、设备挂起、智能频率、不合格后续策略、门店营业状态、场景设备、抓拍、Period/图片写入和模型调用。

Period 先插入取得 `inspectionPeriodId`，图片随后绑定。异步设备任务写 `enterprise_device_capture_info`。

### 5.4 模型

模型配置来自快捷检查项绑定的企业模型配置。代码支持同步单图、批量图片、异步模型任务、设备回调后检测和部分失败重检。

周期业务结果按单图明细和抓拍状态重新聚合，模型 summary 只保存在 `ai_result_json` 用于审计。

## 6. 结果口径

### 6.1 图片

| 字段 | 含义 |
|---|---|
| `aiStatus` | 抓拍和模型执行状态 |
| `aiResult` | AI 原始 `PASS/FAIL/INAPPLICABLE` 或异常结果 |
| `finalResult` | 当前最终图片结果，人工标注后可变化 |
| `aiConfidence` | 模型置信度 |
| `aiReason` | 判断原因 |
| `aiCropHint` | 区域提示 |
| `manualTag*` | 图片人工标注结果、备注、人员和时间 |

### 6.2 Period

| 字段 | 含义 |
|---|---|
| `aiPeriodResult` | 图片聚合后的 AI 结果 |
| `finalResult` | 人工复核后的最终结果 |
| `finalConfidence/finalReasons` | 分流置信度和原因 |
| `aiResultJson` | 批量或原始结果审计信息 |
| `reviewStatus/reviewResult` | Period 复核状态和结果 |
| `ticketId/ticketStatus` | 父 Question 引用和状态快照 |
| `isMisreport/negSampleId` | 误报和负样本引用 |

`calculatePeriodResult()` 的代码规则：

1. 图片为空：`INAPPLICABLE`；
2. 全部设备抓拍失败：`ERROR`；
3. 任一图片 `FAIL`：`FAIL`；
4. 否则任一图片 `PASS`：`PASS`；
5. 否则存在 `INAPPLICABLE`：`INAPPLICABLE`；
6. 其他：`INAPPLICABLE`。

## 7. 人工复核和 Question

只有 Period AI 结果为 `FAIL` 且策略存在扩展配置时进入复核或发单分流。

`createReviewTodo()`：

- 校验具体人员具备门店权限；
- 将岗位解析为门店实际人员；
- 为每人生成待办并发送现有工作通知；
- 未解析到人员时只记录警告，不回退管理员。

`reviewConfirm()`：

- `reviewResult=1`：确认违规，保持 `finalResult=FAIL`，完成待办并发送 Question MQ；
- `reviewResult=2`：标记误报，改为 `finalResult=PASS`，保存负样本，不创建 Question；
- 同一 Period 其他待办一并完成。

`AiInspectionQuestionListener` 创建 `questionType=aiInspection` 的 Question，`businessId=inspectionPeriodId`，图片来自失败图片。Question 链路同步创建 `QUESTION_ORDER`，随后把父 Question ID 和状态回写 Period。

## 9. 事务边界

| 场景 | 写入 | 当前边界 |
|---|---|---|
| 策略维护 | 策略、时段、门店映射 | Service 有 Spring 本地事务 |
| 单店抓拍 | Period、图片、设备任务 | 整体方法无单一事务，夹杂外部调用 |
| 模型结果 | 图片、Period、Redis 频率 | 同步和异步路径分段写入 |
| 复核待办 | 待办、Period、通知 | 未形成单一事务 |
| 人工复核 | Period、待办、负样本、Question MQ | 未标注覆盖全流程的事务 |
| Question | Redis 锁、Question/UnifyTask、Period 引用 | MQ 阶段执行，跨领域分段提交 |
| 状态同步 | Question、Period `ticketStatus` | Question 变化后单独回写 |
| 图片打标 | 图片、Period、待办和可选工单处理 | 方法有事务，仍需核验跨 Service 数据源传播 |

任何一个 Service 方法都不能视为覆盖整条链路的 ACID 事务。

## 10. 一致性风险

### 10.1 重复抓拍

Listener 使用 MQ `messageId` Redis 锁，结束后删除。不同消息 ID 的重复业务任务没有在代码层显示统一的“策略 + 门店 + 抓拍时间”数据库幂等命令。

### 10.2 Period 与图片分步写入

Period 先写、图片后写，中间存在设备和模型调用。异常可能留下空 Period、部分图片或未聚合状态。

### 10.3 Question Redis 锁

部分发单规则先取 Redis 锁再创建 Question。创建失败后锁可能仍阻断后续消息；其他模式又依赖不同业务判断去重。

### 10.4 人工复核并发

`reviewConfirm` 读取 Period 后更新 Period 和全部待办，再异步发送 Question 消息。代码未显示基于版本号的单次确认控制，`ticketId` 又在异步创建后回写，存在并发窗口。

### 10.5 MQ 与数据库

抓拍拆分、单店执行和 Question 创建使用普通消息衔接，未看到事务消息或 Outbox。数据库与消息可能出现分段成功，需要补偿查询和人工恢复。

### 10.6 多租户表名

Mapper 使用 `${enterpriseId}` 拼接表名，不是普通参数绑定。任何新入口都必须从服务端认证和企业配置解析租户后缀，不能接受模型或请求体自由传值。

### 10.7 配置漂移

`EnterpriseModelAlgorithmService.detailByQuickColumnId()` 合并模型默认、企业覆盖和检查项配置。首次执行、异步查询、设备回调和重新检测等路径会重新调用该方法；当前代码未看到 Period 绑定不可变配置快照。

配置在 Period 未完成时变化，后续路径可能使用不同配置。这是代码一致性风险，不代表已发生生产事故。
