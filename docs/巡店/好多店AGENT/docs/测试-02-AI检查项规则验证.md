# AI 检查项规则效果验证测试方案

> **版本**：V0.1
> **状态**：测试基线
> **日期**：2026-07-17

## 1. 测试目标

本文验证 `FR-VALIDATION-001..006`、`AC-VAL-001..007` 及规则同步后的巡检 Period 配置一致性。验证能力是与现有 AI 检查项关联的旁路实验，不改变生产巡检调度、历史结果、复核或工单。

## 2. 测试对象

测试对象包括：

- 数据集与数据集版本；
- 优化集与冻结验收集；
- Baseline 和候选验证版本；
- Prompt、标准图、模型和推理参数快照；
- 逐图结果、模型错误和 Token 用量；
- 指标报告与失败样例；
- 推荐、例外、高风险发布审批；
- 配置同步、前后快照和 Period 生效边界。

## 3. 数据集准备

### 3.1 数据来源

优先从已有 AI 巡店历史抓拍和人工结果中选择图片。原图保留在业务系统或对象存储，验证系统保存图片记录 ID、受控引用、摘要、真值和版本快照；不要求为每次验证重新上传图片。

### 3.2 真值

每张有效图片必须有明确人工真值：

- `PASS`；
- `FAIL`；
- `INAPPLICABLE`，仅用于支持该结果的检查项。

真值由具备对应 AI 检查项管理权限的用户确认。无权限标注、图片损坏、目标不可见或无法形成可信预期的样本不得计入正式验收有效数。

### 3.3 数据集划分

- 优化集：用于误判分析、Prompt 优化、标准图选择和候选生成；
- 验收集：不得参与优化，进入正式验证后冻结。

正式验收集至少 50 张，默认 `PASS >= 15`、`FAIL >= 15`；支持 `INAPPLICABLE` 时该类别至少 5 张，其余样本覆盖真实分布、边界和困难场景。修改成员或真值必须创建新数据集版本，已完成版本继续引用原快照。

## 4. 版本与运行规则

Baseline 使用创建时检查项实际生效配置。每个候选版本只能引用明确的父版本或 Baseline，并固化：

- 数据集版本和成员快照；
- 检查项配置摘要；
- Prompt 和标准图；
- Provider、模型和推理参数；
- 创建人、创建原因和时间；
- 每次实际请求的输入、输出和总 Token；
- 逐图输出、错误、耗时和成本；
- 汇总指标和失败样例。

版本一旦开始正式运行即不可修改。任何 Prompt、标准图、模型、参数、数据集或真值变化都创建新版本。

## 5. 指标口径

| 指标 | 口径 |
|---|---|
| 总体准确率 | 预测与人工真值一致的有效图片数 / 有效图片总数 |
| Precision | 按 `PASS/FAIL/INAPPLICABLE` 分别计算预测命中质量 |
| Recall | 按类别计算真值被正确识别的比例 |
| F1 | 对应类别 Precision 与 Recall 的调和平均 |
| `FAIL` 漏判 | 真值为 `FAIL`，预测为其他业务结果 |
| 误报 | 真值非 `FAIL`，预测为 `FAIL` |
| 模型错误率 | 请求失败、输出无法解析或未形成有效业务结果的图片数 / 正式执行图片数 |
| 相对提升 | 候选准确率减 Baseline 准确率，单位为百分点 |
| 成本 | 逐次 Token、图片调用费用和总运行成本 |

模型错误不得从有效执行总数中静默剔除。报告同时展示总样本数、有效业务结果数和模型错误数。

## 8. 推荐与发布门槛测试

### 8.1 推荐版本

推荐标识要求：

- 冻结验收集完整执行；
- 总体准确率相对 Baseline 提升至少 6 个百分点，或达到检查项目标准确率；
- `FAIL` Recall 不低于 Baseline；
- 模型错误率不超过 2%。

| 用例 | 场景 | 预期结果 |
|---|---|---|
| `TC-VAL-015` | 满足全部推荐条件 | 标记“推荐采用”，但不自动同步生产配置 |
| `TC-VAL-016` | 准确率提升达标但 `FAIL` Recall 下降 | 不推荐，不允许按普通例外同步 |
| `TC-VAL-017` | 准确率达标但模型错误率超过 2% | 不推荐且阻断同步 |
| `TC-VAL-018` | 验收集未完整执行 | 不推荐且阻断同步 |

### 8.2 普通检查项例外

普通版本仅在以下条件全部满足、且只是未达到“提升 6 个百分点或目标准确率”时允许申请例外：

- 验收集完整；
- 总体准确率不低于 Baseline；
- `FAIL` Recall 不低于 Baseline；
- 模型错误率不超过 2%；
- 申请人填写例外原因；
- 另一位具备同等管理权限的用户二次审批。

| 用例 | 场景 | 预期结果 |
|---|---|---|
| `TC-VAL-019` | 仅提升不足，满足其余条件并完成二次审批 | 允许同步并保存原因和两名操作人 |
| `TC-VAL-020` | 申请人与审批人为同一人 | 二次审批无效，不允许同步 |
| `TC-VAL-021` | 准确率低于 Baseline 或 `FAIL` Recall 下降 | 不属于可例外范围，硬阻断 |

### 8.3 高风险检查项

高风险由业务系统确定性配置，至少包括红线、否决和可能直接触发高风险业务动作的检查项，不由模型判断。

高风险同步必须同时满足：

- 正式验收完整；
- 达到配置的绝对目标准确率；
- `FAIL` Recall 不低于 Baseline；
- 模型错误率不超过 2%；
- Baseline 摘要与当前配置一致；
- 候选配置摘要一致；
- 由另一位具备管理权限的用户二次审批。

| 用例 | 场景 | 预期结果 |
|---|---|---|
| `TC-VAL-022` | 满足全部硬门槛但没有二次审批 | 不允许同步 |
| `TC-VAL-023` | 任一硬门槛不满足但审批人尝试豁免 | 仍然阻断，审批不能绕过硬门槛 |
| `TC-VAL-024` | 全部门槛满足且完成二次审批 | 允许进入受控同步 |

## 9. 配置同步测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-VAL-025` | 同步已完成且获准的候选版本 | 只同步白名单 AI 规则字段，保存来源版本、审批和前后快照 |
| `TC-VAL-026` | Baseline 后生产配置被其他人修改 | 当前摘要冲突，阻断同步，要求基于最新配置重新比较 |

| `TC-VAL-028` | 请求夹带结果项、流程、策略时段或门店范围 | 非白名单字段被拒绝，生产配置不变 |
| `TC-VAL-029` | 重复提交相同幂等键和参数 | 返回同一同步结果，不重复修改配置 |
| `TC-VAL-030` | 相同幂等键携带不同参数 | 返回冲突并审计 |
| `TC-VAL-031` | 验证完成但没有用户显式同步 | 生产配置保持不变 |

## 10. Period 生效边界测试

准备两个同步前已创建 Period、两个同步后新建 Period，并覆盖首次调用、异步回调、失败重试、重新检测和周期聚合路径。

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-VAL-032` | 同步前创建 Period，随后同步新配置 | 存量 Period 继续使用原配置快照和摘要 |
| `TC-VAL-033` | 同步后创建新 Period | 新 Period 固化并使用新配置 |
| `TC-VAL-034` | 对存量 Period 执行重试或重新检测 | 所有路径仍读取该 Period 原快照，不读取当前配置 |
| `TC-VAL-035` | 处理异步模型结果、设备回调和周期聚合 | 全链路可还原同一 Period 实际使用配置 |
| `TC-VAL-036` | 在一次定时门店拆分过程中切换配置 | 允许切换边界两侧门店使用不同 Period 快照，不要求原子切换 |
| `TC-VAL-037` | 检查历史结果和 Question | 同步不回算、不覆盖历史结果、复核状态或工单 |

首期不增加 `inspection_batch_id`，因此 `TC-VAL-036` 只验证每个 Period 自洽，不验证调度批次内全部门店同时切换。

## 11. 验收证据

每次正式验证至少保留：

- 数据集版本、成员清单、真值和确认人；
- Baseline 与候选版本完整快照及摘要；
- 逐图输入引用、输出、错误、耗时和 Token；
- 指标计算明细和失败样例；
- 推荐、例外或高风险门槛判定；
- 审批人、原因和时间；
- 同步请求、幂等结果和前后配置快照；
- 各 Period 的配置摘要及实际调用证据。

`TC-VAL-001..037` 全部通过且 `AC-VAL-001..007` 无未解决缺陷后，该专项测试才可签字完成。
