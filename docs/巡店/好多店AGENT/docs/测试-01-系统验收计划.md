# 系统验收计划

> **版本**：V0.1
> **状态**：测试基线
> **日期**：2026-07-17

## 1. 测试目标

本文验证[产品需求](需求-01-产品需求.md)中的业务能力、安全边界、业务系统集成和首批灰度门槛。AI 检查项规则效果验证的专项数据和算法测试见[AI 检查项规则验证](测试-02-AI检查项规则验证.md)。

## 2. 环境与实施输入

实施前由用户提供两个不同租户：

```text
pilot_enterprise_id=<待提供>
isolation_enterprise_id=<待提供>
```

测试数据要求：

| 数据 | 要求 |
|---|---|
| 试点租户 | 1 个区域、最多 30 家门店 |
| AI 员工 | 1 名启用的 AI 风险督导、1 名绑定唯一主门店的 AI 店长及有效人工负责人 |
| 风险规则 | 1 至 3 条已有规则，显式 `agentEnabled=true` |
| 隔离租户 | 不启用 Agent，不配置试点规则 |
| 人员 | 至少包含有效店长、责任督导、审批人及缺人反例 |
| 业务对象 | 可创建和查询 Question、任务载体及审批状态 |
| 数据仓库 | 可区分已刷新、未刷新、未来日期和映射缺失样本 |

第二个租户只用于隔离负向测试，不计入 30 店试点，不用于多租户性能压测。

## 3. 测试层次

- 单元测试：状态迁移、Scope 交集、`request_digest`、幂等键、指标和错误映射；
- 契约测试：Tool、Java `/internal/agent/**`、Webhook 和模型 Usage；
- 集成测试：Hologres、Agent MySQL、Java 业务 Service 和对象引用；
- 系统测试：Workflow、Case、人工审批、Question 和恢复闭环；
- 安全测试：跨租户、越权、参数篡改、非法 Tool 和敏感字段；
- 可靠性测试：超时、结果未知、重复消息、并发、重启和下游不可用；
- 灰度验收：按有效统计日和真实 Trigger/Question 样本量验证。

## 5. Scope 与安全测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-SEC-001` | 使用 `pilot_enterprise_id` 请求 `isolation_enterprise_id` 的门店或对象 | 返回 `PERMISSION_DENIED`，无数据泄露，拒绝记录可审计 |
| `TC-SEC-002` | 请求员工 Scope 外的区域或门店 | Tool Gateway 在访问数据源前拒绝 |
| `TC-SEC-003` | 人工发起人权限小于员工 Scope | 实际 Scope 为两者交集 |
| `TC-SEC-004` | 模型或请求体伪造租户、员工、用户或 Scope | 后端可信上下文不被覆盖，调用拒绝并审计 |
| `TC-SEC-005` | 调用未被 Role、Skill 或 Workflow 允许的 Tool | 返回 `TOOL_NOT_ALLOWED` |
| `TC-SEC-006` | 请求超日期范围、超行数或自由 SQL | 请求被拒绝或按契约截断；自由 SQL 不存在 |
| `TC-SEC-007` | 停用员工、显式收回员工 Scope 或使凭证失效 | 不再允许的新 Run 和写动作立即阻断，历史记录保留 |
| `TC-SEC-008` | 创建时选择超出负责人当前权限的 Scope；创建后再变更负责人权限或状态 | 创建请求被拒绝；已保存的员工独立 Scope 不随负责人后续变化 |
| `TC-SEC-009` | 审批后修改动作参数、对象、版本或摘要 | 原审批失效，写动作被拒绝 |
| `TC-SEC-010` | 检查 Webhook 和日志 | 不包含完整 Prompt、原始模型输出、图片内容或 Secret |
| `TC-SEC-011` | AI 店长请求非 `primary_store_id` 门店画像、记忆或排行明细 | Gateway 返回 `PERMISSION_DENIED`，其他门店信息不进入模型、页面、Webhook 或日志 |
| `TC-SEC-012` | 模型伪造 AI 店长的 `store_id` 或要求写入画像/标签 | 后端忽略或拒绝伪造字段，返回 `PERMISSION_DENIED`/`TOOL_NOT_ALLOWED` 并审计 |
| `TC-SEC-013` | 篡改 Java 写动作授权断言或业务命令参数但不重算请求 HMAC | Java 验签失败，不执行写动作 |
| `TC-SEC-014` | 使用过期或重复 `jti` 的授权断言 | Java 返回过期/重放错误，不执行写动作 |
| `TC-SEC-015` | 断言租户、目标门店、请求摘要与实际请求不一致 | 返回 `TENANT_MISMATCH`/`PERMISSION_DENIED`，命令不落成功 |
| `TC-SEC-016` | Python 与 Java 对同一命令契约样例分别计算 `request_digest` | 字段顺序和无意义空白不同不影响结果；非法 `null`、浮点数或未排序无序 ID 集合被拒绝 |
| `TC-SEC-017` | 使用同一 `jti`、不同 `idempotency_key` 再次提交 | 独立断言消费唯一约束返回 `REPLAY_REJECTED`，不创建第二条命令或业务对象 |
| `TC-SEC-018` | 使用新 `jti`、原 `idempotency_key` 合法重试已完成命令 | 新断言消费记录成功落库，按命令幂等返回原业务结果并累计重复次数，不重复执行业务 |

## 12. 灰度验收

### 12.1 第一阶段：ANALYZE_ONLY

本阶段只验收 AI 风险督导主线，范围固定为试点租户、1 个区域、最多 30 家门店、1 名 AI 风险督导和 1 至 3 条规则。AI 店长按 `TC-FUNC-011..018` 独立验收，不计入风险 Trigger/Case 样本。

退出条件：

- 连续至少 7 个有效统计日；
- 至少 20 条有效 Trigger；
- Scope 越权、跨租户返回和重复未关闭 Case 为 0；
- 模型用量记录覆盖率 100%；
- 人工抽查至少 10 个 Case；
- 事实编造为 0，结果可用率至少 80%；
- 无未解决阻断或高风险缺陷；
- 租户管理员和业务负责人共同签字确认。

这里的共同签字是阶段退出闸门，不替代每个 Question 创建前由 AI Employee 人工负责人执行的逐条审批。

### 12.2 第二阶段：CREATE_QUESTION

在相同范围至少运行 7 个有效统计日，并完成至少 10 次人工审批后的 Question 创建。

退出条件：

- 未审批创建为 0；
- 重复 Question 为 0；
- 审批、Java 命令、业务引用和 Case 关联可追溯率 100%；
- 样本不足时延长阶段，不降低数量门槛。

### 12.3 Epic D：AI 店长独立验收

Epic D（AI 店长与门店画像）单独准备数据、执行测试和签署验收，不计入 Epic A 的风险 Trigger、Case 或 Question 样本。至少覆盖：

- 主门店明细只能读取 `primary_store_id`，其他门店只参与后端聚合；
- 直接区域达到 10 家时使用直接区域，少于 10 家时使用上上级区域，且不继续向更上层扩展；
- 排行结果不泄露其他门店名称、ID、排名、指标值或明细；
- 同一员工、主门店和统计周的周报重复执行幂等；
- Store Memory 与 Store Profile 的来源、版本和读取快照可追溯；
- AI 店长不创建 Case、Question、消息或任务，不修改业务状态。

Epic D 的 P0/P1 缺陷只阻断 Epic D 自身退出，不阻断已满足公共底座门槛和 Epic A 退出条件的风险督导上线。

## 14. TC 优先级与 Epic Gate 映射

测试表保持按业务主题组织，优先级和退出归属在本节集中定义。未列入 P0 的用例默认按 P1 执行；P2 仅用于未来不阻断当前 Epic 退出的增强测试。`COMMON` 用例是所有 Epic 共用门槛，`EPIC_A` 至 `EPIC_D` 只阻断对应 Epic。

| 测试用例 | priority | gate | 说明 |
|---|---|---|---|
| `TC-SEC-001..010`, `TC-SEC-013..018` | P0 | COMMON | 租户隔离、后端权限、敏感信息、授权断言、Nonce/jti 重放和幂等安全边界 |
| `TC-RUNTIME-001..003`, `TC-DATA-010` | P0 | COMMON | Runtime 契约和调度扫描事实是所有 Epic 的公共基础 |
| `TC-FUNC-001..003`, `TC-DATA-001..007`, `TC-OBS-001..004`, `TC-WEBHOOK-001..004`, `TC-AUDIT-001`, `TC-REL-001..007` | P1 | COMMON | Workflow、数据就绪、审计、用量、Webhook 和恢复基础能力 |
| `TC-FUNC-004..010`, `TC-ACTION-001..010` | P1 | EPIC_A | AI 风险督导、Case、审批和 Question 闭环 |
| `TC-IMAGE-001..007` | P1 | EPIC_B | 图片辅助判断与人工复核专项 |
| `TC-VAL-001..037` | P1 | EPIC_C | AI 检查项效果验证、配置同步和 Period 快照专项 |
| `TC-FUNC-011..018`, `TC-SEC-011..012`, `TC-DATA-008..009` | P1 | EPIC_D | AI 店长、门店记忆、画像和隐私排行 |

因此，`TC-SEC-011/012` 不属于公共底座，不阻断 Epic A；公共底座 P0/P1 仍对所有 Epic 生效。每个测试执行记录必须保存 `priority`、`gate`、结果和证据引用，阶段退出只检查对应 Gate 的 P0/P1 用例。

## 15. 准入与退出

测试开始前必须完成接口契约评审、测试租户隔离、必要业务数据准备和可回滚的测试配置。退出按以下独立闸门判断：

- **公共底座准入**：`gate=COMMON` 的 P0/P1 用例通过；该门槛对所有 Epic 生效。
- **Epic A 独立退出**：`gate=EPIC_A` 的 P0/P1 用例和 12.1、12.2 阶段条件通过；Epic A 自身缺陷不影响其他 Epic。
- **Epic B 独立退出**：`gate=EPIC_B` 的 P0/P1 用例通过；Epic B 自身缺陷不影响其他 Epic。
- **Epic C 独立退出**：`gate=EPIC_C` 的 P0/P1 用例通过；Epic C 自身缺陷不影响其他 Epic。
- **Epic D 独立退出**：`gate=EPIC_D` 的 P0/P1 用例和 12.3 通过；Epic D 缺陷不阻断 Epic A，但未通过时不得宣称 Epic D 已验收。

每个退出闸门都必须保留日志、截图、数据库记录、业务对象 ID 和签字记录等验收证据，追踪矩阵不得有空项。
