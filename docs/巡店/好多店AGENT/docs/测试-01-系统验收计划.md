# 系统验收计划

> **版本**：V0.1
> **状态**：测试基线
> **日期**：2026-07-17

## 1. 测试目标

本文验证[产品需求](需求-01-产品需求.md)中的业务能力、安全边界、业务系统集成和首批灰度门槛。AI 检查项规则效果验证的专项数据和算法测试见[AI 检查项规则验证](测试-02-AI检查项规则验证.md)。

## 2. 环境与实施输入

实施前由用户提供两个不同租户：

```text
pilot_enterprise_id=<待提供>
isolation_enterprise_id=<待提供>
```

测试数据要求：

| 数据 | 要求 |
|---|---|
| 试点租户 | 1 个区域、最多 30 家门店 |
| AI 员工 | 1 名启用的 AI 风险督导及有效人工负责人 |
| 风险规则 | 1 至 3 条已有规则，显式 `agentEnabled=true` |
| 隔离租户 | 不启用 Agent，不配置试点规则 |
| 人员 | 至少包含有效店长、责任督导、审批人及缺人反例 |
| 业务对象 | 可创建和查询 Question、任务载体及审批状态 |
| 数据仓库 | 可区分已刷新、未刷新、未来日期和映射缺失样本 |

第二个租户只用于隔离负向测试，不计入 30 店试点，不用于多租户性能压测。

## 3. 测试层次

- 单元测试：状态迁移、Scope 交集、参数摘要、幂等键、指标和错误映射；
- 契约测试：Tool、Java `/internal/agent/**`、Webhook 和模型 Usage；
- 集成测试：Hologres、Agent MySQL、Java 业务 Service 和对象引用；
- 系统测试：Workflow、Case、人工审批、Question 和恢复闭环；
- 安全测试：跨租户、越权、参数篡改、非法 Tool 和敏感字段；
- 可靠性测试：超时、结果未知、重复消息、并发、重启和下游不可用；
- 灰度验收：按有效统计日和真实 Trigger/Question 样本量验证。

## 4. 功能测试

| 用例 | 前置与操作 | 预期结果 |
|---|---|---|
| `TC-FUNC-001` | 创建 Role 下的 AI 风险督导，配置 Scope、负责人和 Skill | 员工获得独立 `employee_id`；配置可查询，不能编辑平台 Definition |
| `TC-FUNC-002` | 使用定时、人工和数仓刷新场景启动 Workflow | 产生唯一实例并固化 Definition、员工、Role、Skill 和 Scope 快照 |
| `TC-FUNC-003` | 让 AI Step 正常执行并产生一次重试 | Step 与多个 Run 关系正确，结构化输出满足 Schema |
| `TC-FUNC-004` | 同一门店、规则和 Workflow 连续多日命中 | 日记录分别保留，只归并到一个未关闭 Case |
| `TC-FUNC-005` | 关闭 Case 后再次命中 | 创建复发 Case，并关联前序 Case |
| `TC-FUNC-006` | 业务对象完成且连续 3 个有效统计日未命中 | 重新查询事实后关闭 Case，保存确定性关闭证据 |
| `TC-FUNC-007` | 数据未刷新或只有模型“建议关闭” | Case 不关闭，记录阻断原因和下次检查时间 |
| `TC-FUNC-008` | 运行 `ANALYZE_ONLY` | 只产生分析、Case Event 和复查计划，不创建业务对象 |
| `TC-FUNC-009` | 运行 `CREATE_QUESTION` 并通过审批 | 创建或返回同一 Case 的唯一 Question 和任务载体引用 |
| `TC-FUNC-010` | 生成催办和消息建议 | 只形成草稿，不调用主动催办或独立消息发送能力 |

## 5. Scope 与安全测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-SEC-001` | 使用 `pilot_enterprise_id` 请求 `isolation_enterprise_id` 的门店或对象 | 返回 `PERMISSION_DENIED`，无数据泄露，拒绝记录可审计 |
| `TC-SEC-002` | 请求员工 Scope 外的区域或门店 | Tool Gateway 在访问数据源前拒绝 |
| `TC-SEC-003` | 人工发起人权限小于员工 Scope | 实际 Scope 为两者交集 |
| `TC-SEC-004` | 模型或请求体伪造租户、员工、用户或 Scope | 后端可信上下文不被覆盖，调用拒绝并审计 |
| `TC-SEC-005` | 调用未被 Role、Skill 或 Workflow 允许的 Tool | 返回 `TOOL_NOT_ALLOWED` |
| `TC-SEC-006` | 请求超日期范围、超行数或自由 SQL | 请求被拒绝或按契约截断；自由 SQL 不存在 |
| `TC-SEC-007` | 停用员工、回收 Scope 或使凭证失效 | 新 Run 和写动作立即阻断，历史记录保留 |
| `TC-SEC-008` | 负责人失效或权限不能覆盖员工 Scope | 暂停新调度和写动作，不自动选择其他人员 |
| `TC-SEC-009` | 审批后修改动作参数、对象、版本或摘要 | 原审批失效，写动作被拒绝 |
| `TC-SEC-010` | 检查 Webhook 和日志 | 不包含完整 Prompt、原始模型输出、图片内容或 Secret |

## 6. 数据与 Tool 测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-DATA-001` | 查询正常 Hologres 风险数据 | 返回租户过滤、统计日期、刷新状态、证据和受控结果大小 |
| `TC-DATA-002` | 构造未来日期、未刷新或映射缺失数据 | 显示数据质量提示，禁止据此执行写动作 |
| `TC-DATA-003` | 查询 Question、任务、审批和复核当前状态 | 结果来自 Java 权威查询，并带业务对象引用 |
| `TC-DATA-004` | 触发成功、失败、超时、越权和被拦截 Tool | 每次调用均记录输入摘要、状态、耗时和错误码 |
| `TC-DATA-005` | 让 Tool 输出不满足 Schema 或超出大小限制 | 输出不进入模型结论或 Workflow 路由，错误可追踪 |

## 7. Question 与人工审批测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-ACTION-001` | 创建风险 Question | 目标语义为 `agentRisk`；适配期映射 `common` 时同时保存请求和实际类型 |
| `TC-ACTION-002` | 查询店长和当前责任督导后创建 | 店长整改、当前督导审批，两个节点默认 `approveType=any`，无默认抄送 |
| `TC-ACTION-003` | 缺少店长或当前责任督导 | 进入 `WAITING_MANUAL_ASSIGNMENT`，不回退管理员、历史督导或通知人 |
| `TC-ACTION-004` | 同一 Case 跨 Run 重复提交 | 使用 `agent-question:{enterpriseId}:{caseId}:v1` 返回同一业务对象 |
| `TC-ACTION-005` | 同一幂等键携带不同参数 | 返回幂等冲突，不创建第二个对象 |
| `TC-ACTION-006` | Java 返回失败或结果未知后重试 | 原键只查询命令结果，不直接再次执行业务 |
| `TC-ACTION-007` | 校验 Question 描述 | 事实、来源和证据摘要由后端生成，模型内容只进入整改建议 |
| `TC-ACTION-008` | 分别配置三个 SLA | 创建审批、整改截止和 Case 复查时间独立生效 |

## 9. 用量、Webhook 与审计测试

| 用例 | 操作 | 预期结果 |
|---|---|---|
| `TC-OBS-001` | 执行多轮模型 Tool Loop、最终汇总、逐图分析和重试 | 每次实际请求都有独立输入、输出、总 Token、耗时和用途记录 |
| `TC-OBS-002` | Provider 不返回 Usage | Token 字段为空，缺失原因有值，不进行估算 |
| `TC-OBS-003` | 显式触发跨模型降级 | 保存原模型、目标模型、触发原因和各次用量 |
| `TC-WEBHOOK-001` | Workflow 首次进入 `COMPLETED` | 使用固化员工配置异步发送一次 `workflow.completed` |
| `TC-WEBHOOK-002` | Webhook 返回超时或错误 | 5 秒内结束，失败只写日志，不重试、不改变 Workflow/Case |
| `TC-WEBHOOK-003` | Workflow 进入失败、阻断或取消 | 不发送 Webhook |
| `TC-WEBHOOK-004` | 未配置或停用员工 Webhook | 跳过投递并记录原因 |
| `TC-AUDIT-001` | 回放一个完整 Case | 可还原员工、触发人、审批人、Run、Tool、证据、业务命令和最终结果 |

## 10. 可靠性与恢复测试

| 用例 | 故障注入 | 预期结果 |
|---|---|---|
| `TC-REL-001` | Workflow Step 执行中重启 Agent Service | 从持久化状态恢复，不重复完成已提交步骤 |
| `TC-REL-002` | 同一 Trigger 并发到达 | 只产生一个有效 Trigger 处理和一个未关闭 Case |
| `TC-REL-003` | 同一 Workflow 被两个 Worker 获取 | 租约或并发控制保证只有一个有效执行者 |
| `TC-REL-004` | Java 在提交后响应超时 | 按幂等键查到原业务结果，不重复创建 Question |
| `TC-REL-005` | Hologres 或 Java 暂时不可用 | Case 保持可恢复状态，失败不被解释为业务完成 |
| `TC-REL-006` | Scope 在等待审批期间收缩 | 执行动作前重新校验并阻断越权写入 |
| `TC-REL-007` | 修改代码中的 Workflow YAML 后恢复旧实例 | 旧实例继续使用自身 Definition 快照 |

## 11. 灰度验收

### 11.1 第一阶段：ANALYZE_ONLY

执行范围固定为试点租户、1 个区域、最多 30 家门店、1 名员工和 1 至 3 条规则。

退出条件：

- 连续至少 7 个有效统计日；
- 至少 20 条有效 Trigger；
- Scope 越权、跨租户返回和重复未关闭 Case 为 0；
- 模型用量记录覆盖率 100%；
- 人工抽查至少 10 个 Case；
- 事实编造为 0，结果可用率至少 80%；
- 无未解决阻断或高风险缺陷；
- 租户管理员和业务负责人共同签字确认。

### 11.2 第二阶段：CREATE_QUESTION

在相同范围至少运行 7 个有效统计日，并完成至少 10 次人工审批后的 Question 创建。

退出条件：

- 未审批创建为 0；
- 重复 Question 为 0；
- 审批、Java 命令、业务引用和 Case 关联可追溯率 100%；
- 样本不足时延长阶段，不降低数量门槛。
