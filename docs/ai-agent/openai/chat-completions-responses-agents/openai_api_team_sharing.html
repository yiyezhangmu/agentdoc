OpenAI Chat Completions、Responses API 与 Agents SDK

从传统消息式调用，到新一代统一响应接口，再到 Agent 运行框架：理解三者层次、历史对话、工具调用、参数差异与工程选型。

Chat Completions

传统消息式模型接口，兼容生态最好，历史和工具循环主要由应用自行管理。

Responses API

OpenAI 新一代统一模型接口，围绕 Item 组织推理、工具、多模态和上下文。

Agents SDK

建立在模型接口之上的 Agent 运行框架，管理工具循环、Session、Tracing 和编排。

Chat Completions

以 messages 为中心，适合传统聊天、简单工具调用和多厂商兼容。

统一接口

Responses API

以 input / output Item 为中心，适合推理、工具、多模态和 Agent 场景。

负责 Agent Loop、工具执行、Session、Guardrail、Handoff 与 Tracing。

最重要的结论：Responses API 是底层模型能力，Agents SDK 是上层 Agent 运行框架；Chat Completions 仍是第三方模型兼容最广的接口。
02 · Chat Completions

Chat Completions API：围绕 messages 工作

Chat Completions 的核心，是将模型需要知道的全部上下文按顺序放入 messages。

基础调用
from openai import OpenAI

client = OpenAI()

response = client.chat.completions.create(
    model="gpt-5.6",
    messages=[
        {
            "role": "developer",
            "content": "你是一个专业的业务分析助手。"
        },
        {
            "role": "user",
            "content": "查询最近7天风险门店。"
        }
    ]
)

print(response.choices[0].message.content)
Chat Completions 不会自动读取你的数据库历史。应用必须把模型需要知道的 developer、user、assistant、tool 消息重新发送。
03 · Roles

五种常见消息角色

developer
开发团队给模型设定的稳定规则：身份、业务口径、输出格式、工具规则和安全边界。
system
传统系统指令角色。第三方 OpenAI-compatible 接口中仍很常见；新 OpenAI 模型通常优先使用 developer。
user
用户输入。属于不可信内容，不能仅依赖 Prompt 实现权限控制。
assistant
模型此前的普通回答，或者模型发起的 tool_calls。
tool
应用执行工具后返回给模型的结果，通过 tool_call_id 与前面的调用关联。

developer 示例

{
  "role": "developer",
  "content": """
你是企业经营数据分析助手。

规则：
1. 使用中文回答。
2. 业务数据必须通过工具查询。
3. 不允许编造数据。
4. 先给结论，再给依据。
"""
}

assistant 工具调用示例

{
  "role": "assistant",
  "content": null,
  "tool_calls": [
    {
      "id": "call_001",
      "type": "function",
      "function": {
        "name": "query_risk_stores",
        "arguments": "{\"days\":7}"
      }
    }
  ]
}

05 · Tool Calling

工具定义与工具调用

工具定义只是告诉模型“系统里有哪些能力、参数怎么传”，并不会自动执行数据库查询。

tools = [
  {
    "type": "function",
    "function": {
      "name": "query_risk_stores",
      "description": "查询指定日期范围内的风险门店。",
      "strict": true,
      "parameters": {
        "type": "object",
        "properties": {
          "start_date": {
            "type": "string",
            "description": "开始日期，YYYY-MM-DD"
          },
          "end_date": {
            "type": "string",
            "description": "结束日期，YYYY-MM-DD"
          },
          "risk_level": {
            "type": "string",
            "enum": ["low", "medium", "high", "all"]
          },
          "limit": {
            "type": "integer",
            "minimum": 1,
            "maximum": 100
          }
        },
        "required": ["start_date", "end_date", "risk_level", "limit"],
        "additionalProperties": false
      }
    }
  }
]

06 · Parameters

Chat Completions 常用参数

参数作用建议
model指定模型先确认模型是否支持工具、结构化输出、temperature、stop 等能力。
temperature控制随机性数据分析和工具调用通常使用 0～0.3；新推理模型未必支持。
top_p核采样通常与 temperature 二选一调整。
stop停止序列不是敏感词过滤；部分新模型不支持。
max_completion_tokens限制最大输出 Token不等于字符数，推理模型还可能包含推理 Token。
tool_choice控制是否及如何使用工具常见值：auto、none、强制指定某个工具。
parallel_tool_calls允许并行工具调用适合读取类操作，写操作要谨慎。
n生成多个候选答案Agent 场景通常固定为 1。
stream流式输出工具参数可能分块返回，需要完整拼接后再解析。
response_format结构化输出优先使用 JSON Schema。
store控制平台存储企业应用按合规要求设置。
metadata非敏感跟踪字段不要放 API Key、Token、密码和个人敏感信息。

temperature

越低越稳定，越高越发散，但低 temperature 不等于高正确率。

temperature=0.2

stop sequence

更准确的名称是“停止序列”，不是敏感词过滤。

stop=["<END>", "【回答结束】"]
07 · Responses API

Responses API：新一代统一模型接口

from openai import OpenAI

client = OpenAI()

response = client.responses.create(
    model="gpt-5.6",
    instructions="你是一个专业的业务分析助手。",
    input="查询最近7天风险门店。"
)

print(response.output_text)
messages → input输入不再局限于传统消息列表。
developer/system → instructions稳定规则有了独立入口。
choices → output输出由类型化 Item 组成。
08 · Item Model

Responses API 的核心：Item

Chat Completions 主要围绕 Message；Responses API 主要围绕 Item。

reasoning

推理相关输出或摘要。

function_call

模型发起函数调用。

message

最终对用户可见的回答。

web_search_call

托管网页搜索调用。

file_search_call

文件检索调用。

computer_call

计算机操作类调用。

{
  "output": [
    {
      "type": "reasoning"
    },
    {
      "type": "function_call",
      "name": "query_risk_stores",
      "call_id": "call_001",
      "arguments": "{\"days\":7}"
    }
  ]
}
Responses API 不只是把参数改名，而是把模型执行过程拆成多个类型明确的 Item。
09 · Conversation State

Responses API 的多轮上下文

方式一：手工传递 input 历史

response = client.responses.create(
    model="gpt-5.6",
    instructions="你是业务分析助手。",
    input=[
        {"role": "user", "content": "查询高风险门店"},
        {"role": "assistant", "content": "共有3家"},
        {"role": "user", "content": "整改情况呢？"}
    ]
)

方式二：previous_response_id

response2 = client.responses.create(
    model="gpt-5.6",
    instructions="你是业务分析助手。",
    previous_response_id=response1.id,
    input="这些门店整改情况怎么样？"
)
企业应用仍应保存自己的会话、工具调用和审计数据。previous_response_id 更适合作为模型上下文能力，而不是业务数据库的替代品。

11 · Agents SDK

Agents SDK：在模型接口之上的 Agent 运行框架

from agents import Agent, Runner, function_tool

@function_tool
def query_risk_stores(days: int) -> str:
    return "真实业务查询结果"

agent = Agent(
    name="业务分析助手",
    instructions="业务数据必须通过工具查询，禁止编造。",
    tools=[query_risk_stores]
)

result = Runner.run_sync(
    agent,
    "查询最近7天风险门店，并分析主要问题。"
)

print(result.final_output)

Agent Loop

自动管理“模型 → 工具 → 模型”的循环。

Session

支持多轮运行状态与会话管理。

Tracing

追踪模型调用、工具调用和运行链路。

Guardrails

输入、输出和工具执行前后的约束机制。

Handoffs

把任务交给另一个专业 Agent。

MCP / Tools

更方便地接入工具与 MCP 服务。

Agents SDK 不会自动解决多租户、数据权限、审批、幂等、限流、事务和审计。它是 Agent Runtime，不是完整业务平台。

13 · Decision Guide

如何选型

普通聊天 / 多厂商兼容

选择 Chat Completions

适合传统问答、简单工具调用，以及 DeepSeek、通义、智谱等 OpenAI-compatible 接口。

新 OpenAI 应用

优先 Responses API

适合推理、多模态、统一工具、Web Search、File Search、Computer Use 和 MCP。

复杂 Agent

选择 Agents SDK

适合连续调用多个工具、Session、Tracing、Guardrails、Handoff 和多 Agent 编排。

混合架构

OpenAI + 第三方模型

OpenAI 走 Responses API，第三方模型走 Chat Completions Adapter，上层统一由业务服务或 Agents SDK 管理。

最终记住四句话：Chat Completions 是传统聊天接口；Responses API 是新一代统一模型接口；Agents SDK 是 Agent 运行框架；企业权限与业务安全仍由业务系统负责。
Message → Item messages → input tool_calls → function_call Agent Loop → Agents SDK 权限边界 → 业务系统
